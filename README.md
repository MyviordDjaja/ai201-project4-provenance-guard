# Provenance Guard

A backend service that any creative-sharing platform can plug into to classify whether submitted text was written by a human or generated by AI, score how confident that classification is, surface a plain-language transparency label to readers, and handle appeals from creators who believe they were misclassified.

The design philosophy throughout is that **a false positive (calling a real human's work AI) is worse than a false negative**. The scoring, the thresholds, and the label wording are all deliberately biased toward protecting writers, and every contested decision has an appeal path to a human.

The full design document, written before any code, is in [planning.md](planning.md).

---

## Quick start

```bash
# from the repo root, with the virtualenv active
pip install -r requirements.txt          # flask, flask-limiter, groq, python-dotenv
echo "GROQ_API_KEY=your_key_here" > .env  # Signal 1 uses the Groq free tier
python app.py                             # serves on http://localhost:5000
```

Submit a piece of text:

```bash
curl -s -X POST http://localhost:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "The sun dipped below the horizon...", "creator_id": "test-user-1"}' \
  | python -m json.tool
```

## Architecture overview: the path a submission takes

A single piece of text flows through the system as follows:

1. The text arrives at **`POST /submit`** (Flask API layer).
2. The **rate limiter** (Flask-Limiter) checks the caller's IP against the configured limits and rejects floods with `429` before any detection work runs, so abuse cannot burn the Groq quota.
3. **Input validation** confirms the text is a non-empty string and flags text shorter than 40 characters as `low_evidence`.
4. The text fans out to **two independent detection signals**: Signal 1 (the Groq LLM) and Signal 2 (Python stylometry). Each returns a `p_ai` in `[0, 1]` (0 = looks human, 1 = looks AI).
5. The **confidence scorer** blends the two signals into a single calibrated `p_ai`, derives a `confidence` value, and assigns a `verdict` of `human`, `uncertain`, or `ai`.
6. The **label builder** turns that verdict and confidence into the reader-facing transparency label text.
7. The full decision is written to the **SQLite audit log**.
8. The API returns a structured JSON response with the verdict, confidence, per-signal breakdown, and label.

The appeal path reuses the same audit log: **`POST /appeal`** looks up an existing decision by `content_id`, writes an appeal entry referencing the original decision, flips the content's status to `under_review`, and returns confirmation. There is no automated re-classification; an appeal flags the item for a human.

```
                          POST /submit
                               |
                        rate limit (429?)
                               |
                        input validation
                               |
                 +-------------+-------------+
                 |                           |
          Signal 1: LLM              Signal 2: stylometry
          (Groq 70b)                 (pure Python)
            p_ai, rationale            p_ai, features
                 |                           |
                 +-------------+-------------+
                               |
                   confidence scorer (asymmetric)
                    -> p_ai, confidence, verdict
                               |
                        label builder (3 variants)
                               |
                    audit log (SQLite)  <----  POST /appeal
                               |                  (status -> under_review)
                        JSON response
```

(A more detailed two-flow diagram is in [planning.md](planning.md) under `## Architecture`.)

## Detection signals

The pipeline uses two distinct signals chosen because **they fail in different ways**: the LLM understands meaning but is poorly calibrated, while the stylometry understands surface form but is blind to meaning. Combining them is more robust than either alone.

### Signal 1 - LLM classifier (Groq, `llama-3.3-70b-versatile`)

- **What it measures:** a holistic, semantic judgement of whether the passage reads as AI-generated (coherence that is too even, generic phrasing, lack of lived specificity, formulaic structure). It is prompted to be cautious and calibrated, returns `{p_ai, rationale}` as JSON, and is run at `temperature=0`.
- **Why I chose it:** large models have absorbed enormous amounts of generated text and pick up on tone and structure cues that pure statistics miss. It reasons about content, not just form.
- **What it misses:** it is not a calibrated detector. It can be confidently wrong, it is known to be biased against non-native English writers, it can be defeated by lightly edited AI text, and its output is mildly non-deterministic even at `temperature=0`. This is exactly why it is never trusted alone.

### Signal 2 - Stylometric heuristics (pure Python, no external libraries)

It computes five surface features, maps each to a per-feature AI-likelihood, and blends them:

| Feature | What it measures | AI tendency | Weight |
|---|---|---|---|
| Burstiness | coefficient of variation of sentence length | low (uniform) | 0.30 |
| Connective density | formulaic connectives per sentence ("moreover", "furthermore", "it is important to note") | high | 0.30 |
| Punctuation variety | distinct rich punctuation marks used | low | 0.15 |
| Lexical diversity | type-token ratio | low | 0.10 |
| Repetition | fraction of repeated word bigrams | high | 0.15 |

- **Why I chose it:** decoding strategies push generated text toward the statistical center (even sentence lengths, smooth vocabulary, formulaic connectives), whereas human writing is statistically burstier and messier. It is also free, fast, deterministic, and explainable, which balances the LLM's opacity.
- **What it misses:** it is purely formal and ignores meaning. It mislabels naturally uniform human writing (academic prose, formal essays, non-native English) as AI, it is noisy on short text, and it is easily gamed by deliberately adding variance.

## Confidence scoring

### How the two signals are combined

The design intent came first: `p_ai` is a probability in `[0, 1]` where **0.5 means the system genuinely cannot tell**, and `confidence = 2 * |p_ai - 0.5|` measures how far from a coin-flip we are. This makes "uncertain" a first-class outcome instead of a rounding error.

The combination (in `scoring.py`) does three things, all driven by the false-positive asymmetry:

1. **Weighted blend:** `blend = 0.6 * llm + 0.4 * stylometry`. The LLM reasons about meaning and carries more weight; stylometry is the cheaper formal guard against the LLM's miscalibration.
2. **Disagreement shrinks toward 0.5:** the larger the spread between the two signals, the more `p_ai` is pulled back toward 0.5. This is how "a confident AI call requires both signals to agree" is enforced without a hard gate.
3. **Asymmetric verdict bands:** `ai` requires `p_ai >= 0.70`, `human` requires `p_ai <= 0.40`, everything between is `uncertain`. It deliberately takes more evidence to accuse than to clear. A low-evidence (short) submission has its deviation from 0.5 capped, so short text can never reach the AI band.

| p_ai range | verdict | label variant |
|---|---|---|
| `<= 0.40` | `human` | Likely human-written |
| `0.40 - 0.70` | `uncertain` | Origin uncertain |
| `>= 0.70` | `ai` | Likely AI-generated |

### How I validated it is meaningful

I ran a calibration set spanning the confidence range (clearly AI, clearly human, formal-human borderline, lightly-edited-AI borderline) plus band checks for a blatant AI sample and a very short sample. Scores varied sharply and landed in the intuitive bands, and all three verdicts were reachable. Two representative real outputs from the running system:

**High-confidence example (creator "maya", casual restaurant review):**

```
verdict:    human
p_ai:       0.101
confidence: 0.798   ("Likely human-written - Confidence: 80%")
signals:    llm = 0.10   stylometry = 0.05   (both agree -> high confidence)
```

**Lower-confidence example (creator "jordan", formal AI-ethics paragraph):**

```
verdict:    uncertain
p_ai:       0.649
confidence: 0.297   ("Origin uncertain - Confidence: 30%")
signals:    llm = 0.70   stylometry = 0.609  (leaning AI, but not enough to commit)
```

The same scorer produces a confident, reassuring human verdict in one case and an honest "we cannot tell" in the other. The casual review scores `0.10` because both signals strongly agree it is human; the formal paragraph reaches only `0.649` (below the `0.70` AI bar) because, even though both signals lean AI, the asymmetric threshold refuses to accuse a borderline case. That gap (`0.80` vs `0.30` confidence) is the meaningful variation the scoring is designed to produce.

## Transparency label

The label is what a non-technical reader sees on the platform. There are three variants selected by verdict; the confidence percentage is folded into the headline. The AI variant is deliberately non-accusatory and the human variant deliberately reassuring, reflecting the false-positive asymmetry. The exact text each variant displays:

### Variant A - high-confidence AI (`verdict = ai`)

> **Likely AI-generated - Confidence: NN%**
>
> Our automated analysis suggests this content was most likely produced with the help of an AI text generator. This is an estimate, not a verified fact, and it can be wrong. If you are the creator and believe this is a mistake, you can appeal and a human will review it.

### Variant B - high-confidence human (`verdict = human`)

> **Likely human-written - Confidence: NN%**
>
> Our automated analysis found no strong signs that this content was generated by AI. It appears to be the creator's own work. This is an estimate and not a guarantee of authorship.

### Variant C - uncertain (`verdict = uncertain`)

> **Origin uncertain - Confidence: NN%**
>
> Our automated analysis could not confidently determine whether this content was written by a person or generated by AI. Please treat its origin as unverified. If you are the creator, you can appeal to add context and have a human review it.

(`NN%` is the live confidence value; e.g. the two examples above rendered as "Confidence: 80%" and "Confidence: 30%".)

## Appeals workflow

- **Who can appeal:** the creator of the content, keyed by `content_id`. This prototype has no auth, so any caller holding the `content_id` can appeal; a real deployment would gate this to the authenticated owner.
- **What they provide:** `content_id` (must match a prior decision) and `creator_reasoning` (free-text explanation).
- **What the system does:** validates the `content_id` exists (else `404`), flips the original content's status to `under_review`, appends an `appeal` event to the audit log that stores the reasoning verbatim and back-references the original decision id, and returns the `appeal_id` and new status. **No automated re-classification** - the appeal flags the item for a human, which is the honest behavior given that AI detection is not reliable enough to auto-overturn.

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-CONTENT-ID", "creator_reasoning": "I wrote this myself. I am a non-native English speaker and my writing may read as more formal than typical."}'
```

```json
{
  "content_id": "661eaa0b708b421b8bd86a711215a49e",
  "appeal_id": "4c1830a312db4400bb3b0417dca0a87b",
  "status": "under_review",
  "message": "Appeal received. The classification is now under review by a human."
}
```

A human reviewer opening the queue sees, for each appeal, the original decision in full (verdict, p_ai, confidence, both signal scores and features, and the label shown) alongside the creator's reasoning - which is what lets them judge whether, for example, a non-native English writer was unfairly flagged by the stylometry signal.

## Rate limiting

Implemented with Flask-Limiter, keyed on client IP, in-memory storage (`storage_uri="memory://"`).

| Endpoint | Limit | Reasoning |
|---|---|---|
| `POST /submit` | **10 per minute; 100 per day** | A real writer submits a handful of pieces in a sitting, so 10/min comfortably covers legitimate bursts (edits, resubmits) while a flooding script hits the wall immediately. 100/day caps sustained scraping of the classifier and protects the Groq free-tier quota. |
| `POST /appeal` | **5 per minute; 20 per day** | Appeals are rare, deliberate actions. A high appeal rate signals abuse, so the limit is tighter than for submissions. |

These numbers are a defensible starting point, not a hard claim of optimality; in production they would be tuned against real traffic and the limiter would point at Redis instead of in-memory storage (in-memory counters reset on restart and are per-process).

**Evidence** - 12 rapid requests against the 10/min limit (first 10 pass, rest are rejected):

```
request 1  -> HTTP 200
request 2  -> HTTP 200
...
request 10 -> HTTP 200
request 11 -> HTTP 429
request 12 -> HTTP 429
```

Each entry captures timestamp, content id, creator id, attribution, combined `p_ai` and `confidence`, **both** individual signal scores, the label variant, and a `detail` blob (rationale, full stylometry feature vector, signal spread; for appeals, the reasoning and the original decision id).

## Known limitations

**Formal or non-native human writing is the most likely misclassification, and it is the worst kind.** The stylometry signal flags low burstiness and high connective density as AI markers - but those are exactly the properties of fluent academic prose, legal writing, and fluent-but-non-native English. In the calibration runs, a formal human-style paragraph on monetary policy raised the stylometry score to `0.379` (well above the `0.05` a casual review scores, though still short of the `0.5` midpoint), and the AI-ethics paragraph reached `p_ai 0.649` partly on its connective density (`connective_ratio` 0.667, `connective_ai` 1.0). The asymmetric `0.70` threshold and the requirement that both signals agree keep these cases out of the `ai` band (they land `uncertain` or `human`), and the appeal path exists specifically for the writers this would otherwise harm - but the underlying signal weakness is real and tied directly to how stylometry works.

Two further limitations worth naming honestly:

- **Lightly edited / "laundered" AI text** can defeat both signals at once: human edits add the punctuation and sentence-length variance that stylometry keys on, and may also slip past the LLM. The system does not claim to catch this; such text correctly lands in the uncertain or human range rather than being forced to a verdict.
- **The LLM signal is mildly non-deterministic** even at `temperature=0`. The same casual review scored `llm = 0.05` on one run and `0.10` on another, shifting confidence between ~0.80 and ~0.90. The verdict band was stable, but exact scores are not perfectly reproducible.

## Spec reflection

**One way the spec helped:** writing the confidence semantics in `planning.md` section 3 *before* coding - deciding that 0.5 means "cannot tell", that `confidence = 2 * |p_ai - 0.5|`, and that the thresholds would be asymmetric - meant the scorer was a near-direct translation of the spec rather than something I reverse-engineered from code. Following the hint "decide what 0.5 means to a user, then figure out how to achieve it" forced the hardest design decision up front, so by the time I wrote `scoring.py` there were no open questions about what a number should mean.

**One way the implementation diverged:** `planning.md` section 6 predicted the lightly-edited-AI sample would land mid-range `uncertain`. In practice it landed `human` (`p_ai 0.243`) because both signals were genuinely fooled by the human editing. I deliberately did **not** tune the thresholds to force the predicted outcome, because doing so would have been gaming the demo - it would misrepresent the system's real (and limited) ability to catch laundered AI. The honest behavior is to report low confidence and let the appeal/review process handle edge cases, which is what the system now does.

## AI usage

This project was built with AI assistance (Claude), generating the implementation milestone-by-milestone against the `planning.md` spec and the architecture diagram. Two specific instances where the direction or a deliberate decision mattered:

1. **Signal 1 prompt and output handling (`detection.py`).** Rather than accept a generic "detect AI" prompt, I directed the AI to write a cautious, calibrated detection prompt that explicitly tells the model many humans write formally and that non-native English speakers can read as AI, so it should only assign a high probability on strong signals. I also required strict JSON output (`response_format={"type": "json_object"}`) and added a defensive `_clamp01` step so a malformed or out-of-range value from the model falls back to `0.5` rather than corrupting the score.

2. **Keeping the `0.70` AI threshold instead of weakening it.** The `0.70` bar was set in the spec (`planning.md` section 3) to encode the false-positive asymmetry. When the milestone's moderate "clearly AI" calibration sample landed at `p_ai 0.649` (`uncertain`) rather than in the `ai` band, the easy way to make the demo look cleaner would have been to lower the threshold. I kept it at `0.70` and instead verified the `ai` band was genuinely reachable by testing a blatant AI sample, which scored `p_ai 0.72`. Lowering the threshold to flatter the demo would have defeated the point of protecting human writers.

A smaller revision worth noting: the stylometry calibration anchors and weights were chosen deliberately (burstiness and connectives weighted highest at `0.30` each), and I validated that choice by running the calibration set and confirming the formal-human borderline stayed out of the `ai` band (it lands at `p_ai 0.394`, `human`) rather than assuming the weights were right.

## API reference

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/submit` | Classify text. Body: `{ "text": "...", "creator_id": "..." }`. Returns content_id, verdict, p_ai, confidence, per-signal breakdown, and label. Rate limited 10/min, 100/day. |
| `POST` | `/appeal` | Contest a classification. Body: `{ "content_id": "...", "creator_reasoning": "..." }`. Flips status to `under_review`. Rate limited 5/min, 20/day. |
| `GET` | `/log` | Return the structured audit log, newest first. |
| `GET` | `/health` | Liveness check. |
