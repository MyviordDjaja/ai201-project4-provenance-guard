# Provenance Guard - Planning

Backend service that any creative-sharing platform can plug into to classify whether submitted text was written by a human or generated by AI, score how confident that classification is, surface a plain-language transparency label, and let creators appeal a classification they disagree with.

---

## 1. Architecture Narrative

The path a single piece of text takes from submission to the label a reader sees:

1. A creator sends the text to `POST /submit`. The **API layer** (Flask) receives the request.
2. The **rate limiter** (Flask-Limiter) checks the caller against the configured limits. If the caller is over the limit the request is rejected with `429` before any detection work happens, so flooding cannot burn Groq quota.
3. The request passes through **input validation**: the text must be present, be a string, and meet a minimum length. Very short text is flagged as low-evidence, which later widens the uncertain band.
4. The validated raw text is handed to the **detection pipeline**, which runs two independent signals:
   - **Signal 1 - LLM classifier (Groq, llama-3.3-70b-versatile):** returns an AI-likelihood probability plus a short rationale.
   - **Signal 2 - Stylometric heuristics (pure Python):** returns an AI-likelihood probability derived from surface statistics of the text.
   Each signal returns a number in `[0, 1]` where 0 means "looks human" and 1 means "looks AI", plus the raw features it measured.
5. The two signal outputs flow into the **confidence scorer**. It combines them into a single `p_ai` (probability the text is AI), then derives a `confidence` value (how far the system is from a coin-flip) and a `verdict` (one of `human`, `ai`, `uncertain`). The scorer is deliberately asymmetric: it requires more evidence to call something AI than to call it human, because a false AI accusation against a real writer is the worst outcome.
6. The verdict and confidence are passed to the **label builder**, which selects one of three transparency-label variants and fills in the plain-language text that a reader on the platform would see.
7. The complete decision - content id, verdict, `p_ai`, confidence, both signal scores and features, and the label text - is written to the **audit log** (SQLite). This is the canonical, append-only record of every decision.
8. The **API layer** returns a structured JSON response containing the verdict, confidence, the per-signal breakdown, and the transparency label text.

The appeal path reuses the same audit log:

1. A creator who disagrees sends `POST /appeal` with the `content_id` of the original decision and their written reasoning.
2. The **appeal handler** looks up the original decision in the audit log. If it does not exist, it returns `404`.
3. The handler records the appeal (creator reasoning + a back-reference to the original decision) as a new audit-log entry and updates the original content's status to `under_review`.
4. The API returns confirmation, the new status, and the appeal id. Re-running detection automatically is explicitly out of scope; an appeal flags the decision for human review.

## 2. Detection Signals

The pipeline uses two distinct, independent signals. They were chosen because they fail in different ways: the LLM is good at meaning and bad at calibration, the heuristics are good at surface regularity and blind to meaning. Combining them is more robust than either alone.

### Signal 1 - LLM classifier (Groq, llama-3.3-70b-versatile)

- **What it measures:** a holistic, semantic judgement of whether the passage reads as AI-generated - coherence that is too even, hedging, generic phrasing, lack of lived specificity, "as an AI" tells, topic-drift patterns.
- **Why it differs human vs AI:** large models have seen enormous amounts of generated text and can pick up on tone and structure cues that pure statistics miss. It reasons about content, not just form.
- **Blind spot:** it is not a calibrated detector. It can be confidently wrong, it is known to be biased against non-native English writers (flagging them as AI), it can be defeated by lightly edited AI text, and its output is non-deterministic. We never trust it alone, and we never let it single-handedly produce a high-confidence AI verdict.

### Signal 2 - Stylometric heuristics (pure Python, no external libraries)

- **What it measures:** surface statistics of the text:
  - **Burstiness** - variance in sentence length. Human writing mixes long and short sentences; AI tends to be uniform.
  - **Lexical diversity** - type-token ratio. AI text is often smoothly repetitive in vocabulary.
  - **Connective / transition density** - overuse of "moreover", "furthermore", "additionally", "in conclusion", "it is important to note", which AI over-produces.
  - **Punctuation variety** - humans use dashes, ellipses, parentheses, varied punctuation; AI tends toward a uniform comma-period rhythm.
  - **Repetition** - repeated n-grams and sentence-opening words.
  These features are normalized and combined into a single `[0, 1]` AI-likelihood.
- **Why it differs human vs AI:** decoding strategies push generated text toward the statistical center - even sentence lengths, smooth vocabulary, formulaic connectives - whereas human writing is statistically "burstier" and messier.
- **Blind spot:** it is purely formal and ignores meaning. It mislabels naturally uniform human writing (technical docs, formal essays, non-native writers) as AI, it is unreliable on short text where statistics are noisy, and it is easy to game by deliberately adding variance. It cannot tell a true statement from a fabricated one.

## 3. Confidence Scoring and the False-Positive Problem

**Design intent first (per the hint): decide what the score means, then compute it.**

- `p_ai` in `[0, 1]` = probability the text is AI-generated. `0.5` means the system genuinely cannot tell.
- `confidence = 2 * |p_ai - 0.5|`, in `[0, 1]` = how far from a coin-flip we are. A `p_ai` of 0.5 gives confidence 0; a `p_ai` of 0.95 or 0.05 gives confidence 0.9. This makes "uncertain" a first-class outcome rather than a rounding error.

**Combining the signals.** `p_ai` is a weighted blend of the two signals. The weighting and the verdict thresholds are deliberately asymmetric, because on a writing platform a false positive (calling a real human's work AI) is worse than a false negative:

- The dead zone where we refuse to commit is wide and shifted toward the AI side.
- To return an `ai` verdict we require `p_ai >= 0.70`. To return a `human` verdict we only require `p_ai <= 0.40`. Anything in between is `uncertain`.
- A high-confidence `ai` verdict requires *both* signals to agree. If only the LLM screams AI but the stylometry disagrees, the result is pulled toward `uncertain`, never a confident accusation.
- Very short / low-evidence text caps the maximum confidence, so we cannot loudly accuse someone on the basis of two sentences.

**Verdict bands (target behavior):**

| p_ai range           | verdict     | label variant          |
|----------------------|-------------|------------------------|
| `p_ai <= 0.40`       | `human`     | high-confidence human  |
| `0.40 < p_ai < 0.70` | `uncertain` | uncertain              |
| `p_ai >= 0.70`       | `ai`        | high-confidence AI     |

A `0.51` lands in the uncertain band and produces the hedged label; a `0.95` clears the AI bar and produces the confident-AI label. The two are meaningfully different to the reader, which is the requirement.

**How we test the scores are meaningful (planned):** run a small labeled corpus of known-human and known-AI samples through the pipeline and check that known-human samples rarely cross into the `ai` band (low false-positive rate), that obvious AI lands in the `ai` band, and that ambiguous / edited samples land in `uncertain`.

### False-positive trace

A human poet submits a tightly-structured, formal poem.

1. `POST /submit` accepts it.
2. Signal 2 (stylometry) sees uniform line lengths and low lexical variety and reports a high AI-likelihood - a false alarm, exactly its known blind spot.
3. Signal 1 (LLM) reads the imagery and specificity and reports low AI-likelihood.
4. The signals disagree, so the scorer lands `p_ai` near the middle. Because the AI bar (0.70) is high and requires agreement, the verdict is `uncertain`, not `ai`. The human is not falsely accused.
5. The label shown is the uncertain variant: it states the system could not confidently determine origin and explicitly tells the creator they can appeal.
6. If the creator still disagrees, `POST /appeal` records their reasoning, writes it to the audit log next to the original decision, and sets the content status to `under_review` for a human to look at.

This is the asymmetry made concrete: disagreement between signals protects the writer instead of condemning them.

## 4. API Surface (the contract)

### `POST /submit`
Submit text for attribution analysis.

Request:
```json
{ "content_id": "optional-string", "text": "the poem / story / post ..." }
```
Response `200`:
```json
{
  "content_id": "generated-or-echoed-id",
  "verdict": "human | ai | uncertain",
  "p_ai": 0.27,
  "confidence": 0.46,
  "signals": {
    "llm":        { "p_ai": 0.20, "rationale": "..." },
    "stylometry": { "p_ai": 0.34, "features": { "burstiness": 0.7 } }
  },
  "label": { "variant": "human", "title": "...", "body": "..." },
  "status": "classified"
}
```
Errors: `400` invalid/empty text, `429` rate limited.

### `POST /appeal`
Contest a prior classification.

Request:
```json
{ "content_id": "id-from-submit", "reason": "why the creator disagrees" }
```
Response `200`:
```json
{ "content_id": "...", "appeal_id": "...", "status": "under_review" }
```
Errors: `400` missing fields, `404` unknown content_id.

### `GET /log`
Return the structured audit log (most recent first) so graders can see decisions and appeals. Each entry includes timestamp, event type (`decision` / `appeal`), content id, verdict, confidence, signals used, and (for appeals) the reason and a reference to the original decision.

### `GET /health`
Liveness check. Returns `{ "status": "ok" }`.

## 5. Rate Limiting (planned values)

Limits are documented in detail in the README. Planned starting point, sized to realistic single-creator behavior on a writing platform while blocking floods:

- `POST /submit`: a small per-minute burst limit plus a larger per-day limit per client. A genuine creator submits a handful of pieces at a time; an adversary trying to scrape the classifier or burn Groq quota hits a wall quickly.
- `POST /appeal`: a tighter limit, since appealing is rare and high-volume appeals indicate abuse.

Exact numbers and reasoning are finalized in the README.

---

## 6. Audit Log (planned)

SQLite table, append-only. One row per event. Every attribution decision and every appeal is captured with: timestamp, event type, content id, verdict, `p_ai`, confidence, both signal scores and their features, label variant, and - for appeals - the creator's reason and the id of the decision being appealed. Exposed read-only through `GET /log`. At least three entries are shown in the README.
